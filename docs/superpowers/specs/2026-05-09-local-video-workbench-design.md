# 本地内部视频生成工作台设计

日期：2026-05-09

## 目标

把当前 `douyin-finance-video-polisher` skill 扩展成一个本地可用的网站工作台。用户在浏览器里上传音频、填写少量信息、选择财经短视频模板，然后由本机后端调用转写、分镜、HyperFrames/FFmpeg 渲染流程，生成可下载的 9:16 MP4。

第一版只服务内部试用，不做公网平台、账号系统、收费、多租户和云存储。

## 非目标

- 不把 GitHub Pages 静态站直接变成在线生成平台。
- 不承诺多人同时高并发生成。
- 不做公网文件上传和长期托管。
- 不做完整模板市场。
- 不把所有视频质量问题完全自动化修好；第一版只做结构化生成和基础复审证据。

## 推荐架构

采用本地双层结构：

- 前端：一个本地网页工作台，沿用现有 `docs/index.html` 的视觉语言，提供上传、表单、任务进度、结果下载。
- 后端：Python FastAPI 服务，负责接收文件、创建任务、调用能力检测、转写、分镜、渲染和导出。
- 任务执行：本机串行任务队列。第一版一次处理一个任务，避免多任务同时抢 FFmpeg/浏览器资源。
- 文件目录：本地保存上传、任务元数据、渲染中间文件和最终 MP4。

本地访问地址建议为：

```text
http://localhost:8787
```

## 用户流程

1. 用户启动本地工作台。
2. 浏览器打开 `http://localhost:8787`。
3. 页面显示本机能力状态：FFmpeg、Node/npm、HyperFrames、浏览器预览、转写能力。
4. 用户上传音频文件，支持 `.wav`、`.mp3`、`.m4a`。
5. 用户填写：
   - 视频主题
   - 流量词
   - 可选文案
   - 模板风格
   - 输出文件名
6. 点击生成。
7. 后端创建任务并返回任务 ID。
8. 页面轮询任务状态，显示：
   - 已上传
   - 能力检测
   - 转写中
   - 分镜中
   - 渲染中
   - 合成音频
   - 动态复审
   - 完成或失败
9. 完成后显示下载按钮、任务摘要和基础质量检查结果。

## 第一版功能

### 上传与表单

- 单个音频上传。
- 文件大小第一版限制为 200MB。
- 文案可选：如果用户提供文案，优先使用用户文案；如果没有文案，尝试自动转写。
- 模板第一版提供 3 个选项：
  - 技术分析
  - AI 量化
  - 书籍推荐

### 能力检测

复用 `scripts/check_capabilities.py` 的检测逻辑，并在网页中解释缺失项：

- FFmpeg：合成音频、抽帧、检查视频流。
- ffprobe：读取音频/视频时长和流信息。
- Node/npm/npx：运行 HyperFrames 或相关浏览器工具。
- HyperFrames CLI：渲染 HTML 视频。
- 浏览器预览：动态检查转场和音画同步。
- 转写能力：按语义拆分画面段落。

如果能力不足，页面不只显示“缺少工具”，还要说明：

- 缺什么。
- 它在生成视频中负责什么。
- 没有它会影响哪类质量问题。
- 下一步应该先安装什么。

### 任务系统

每个任务保存一个目录：

```text
workbench/jobs/<job_id>/
  input/
  transcript/
  plan/
  render/
  review/
  output/
  job.json
```

`job.json` 保存任务状态、输入参数、错误信息、输出路径和复审摘要。

第一版任务队列串行执行。后续如果搬到服务器，再改为 Redis/RQ、Celery 或云队列。

### 转写和文案

第一版支持两种模式：

- 用户提供文案：直接按文案和音频时长做语义分段。
- 用户不提供文案：尝试调用本机可用转写工具；如果没有转写能力，任务进入 `needs_script` 状态，提示用户补文案。

精准字级时间戳作为增强项，不作为第一版必须条件。

### 视频生成

视频生成遵循 `SKILL.md` 和 `references/production_patterns.md`：

- 画布 1080x1920。
- 遵守抖音底部安全区。
- 不默认加满屏字幕。
- 每个主场景包含标题和 2-4 个支撑视觉元素。
- 场景按音频语义切分，不平均切页。
- 至少使用 3 种财经视觉配方，例如对比卡、回测表、指标图、规则清单、书籍 CTA。

第一版生成质量目标是“稳定生成 80 分以上的内部草稿”，不是完全无人值守交付最终商业片。

### 动态复审

渲染完成后自动生成基础复审证据：

- ffprobe 检查最终 MP4 是否有视频流和音频流。
- 检查时长是否接近音频时长。
- 抽取整片联系表，方便快速定位结构。
- 对最后 10-20 秒生成 motion strip，重点检查 CTA 和末尾音画同步。

如果本机有浏览器/媒体预览能力，页面提供“打开预览”按钮。

## API 设计

### `GET /api/capabilities`

返回本机能力检测结果。

### `POST /api/jobs`

创建生成任务。请求包含音频文件和表单参数。

### `GET /api/jobs`

列出最近任务。

### `GET /api/jobs/{job_id}`

查询任务状态、进度、日志摘要和输出信息。

### `GET /api/jobs/{job_id}/download`

下载最终 MP4。

### `GET /api/jobs/{job_id}/review`

查看复审图、motion strip 和检查摘要。

## 前端页面

工作台首页分为四块：

- 能力状态条：显示当前机器是否适合生成视频。
- 上传区：音频、文案、主题、模板、输出名。
- 任务进度区：显示当前任务阶段和错误提示。
- 最近任务区：下载、重新打开复审、查看失败原因。

界面要保持工具感，不做营销落地页。用户打开第一屏就能上传和生成。

## 错误处理

常见错误要给出可行动解释：

- 没有 FFmpeg：不能合成音频和抽动态复审图。
- 没有 HyperFrames：不能渲染 HTML 视频。
- 没有转写能力且未填文案：需要用户补文案。
- 音频太长：建议拆成多个任务。
- 渲染失败：保留中间文件和错误日志，方便 Codex 继续排查。

## 安全和隐私

第一版默认只监听本机地址 `127.0.0.1`，不暴露到公网。

内部局域网共享可以作为后续开关，但必须默认关闭。开启局域网访问时，至少需要简单访问密码，避免同一网络里其他人随便上传大文件。

上传文件和输出文件只保存在本机。后续可以增加自动清理，例如保留最近 30 个任务或 7 天内文件。

## 测试与验收

第一版完成后至少验证：

- 能打开 `http://localhost:8787`。
- 能上传一个 `.wav` 文件。
- 没有文案且无转写能力时，会提示补文案。
- 有文案时能创建任务。
- 能生成一个 MP4。
- 生成结果有音频流和视频流。
- 输出时长与音频时长接近。
- 页面能下载 MP4。
- 缺少关键工具时，页面能解释缺什么和为什么需要。

## 后续演进

如果内部版跑通，再考虑：

- 批量上传多个音频。
- 模板库和风格预设。
- 手动微调分镜时间。
- 自动二次审片和自动修复。
- 局域网多人使用。
- 搬到云服务器。
- 用户登录、权限、存储和计费。
